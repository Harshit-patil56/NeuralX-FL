--------------------------------------------------------------------------------
0. HOW I BUILT THIS PROJECT  (mentor summary)
--------------------------------------------------------------------------------

MOTIVATION
  I wanted to solve a real healthcare problem: hospitals hold valuable pathology
  data but can never share it due to HIPAA/GDPR. Standard Federated Learning
  looked like the answer — but two gaps stood out: it has no protection against
  malicious hospitals sending poisoned updates, and it stalls when hospitals have
  unequal network speeds. I decided to build a system that closes both gaps at
  once.

STEP 1 — DEFINE THE PROBLEM CLEARLY
  I framed the project as a simulation of 10 hospitals collaboratively training
  a colon-pathology classifier (PathMNIST, 9 classes) without sharing raw data.
  I deliberately assigned different "roles" to hospitals — 20% malicious,
  20% noisy, 20% unreliable — to create a realistic adversarial environment.

STEP 2 — DESIGN THE ARCHITECTURE BEFORE CODING
  I drew out the full pipeline on paper first:
    Data → Non-IID partition (Dirichlet α=0.5) → Local training →
    Byzantine detection → Async selection → Robust aggregation → Global model
  This forced me to identify every decision point before writing a single line.

STEP 3 — BUILD MODULARLY (one file = one responsibility)
  I wrote 8 focused modules so each could be tested and debugged in isolation:
    config.py     — all hyperparameters in one place, no magic numbers anywhere
    model.py      — PathologyNet CNN (2 conv layers + 2 FC layers, GPU-only)
    data.py       — memory-mapped PathMNIST loading (mmap_mode="r")
    client.py     — local training, attack injection, unreliable dropout
    aggregation.py — trimmed_mean, coordinate median, Krum, FedAvg
    detection.py  — L2 norm filter, cosine similarity filter, label noise
    strategy.py   — async buffer, detection pipeline, robust aggregation
    evaluation.py — accuracy/loss tracking, 7 plots
  main.py wires them together and runs 6 experiments in sequence.

STEP 4 — WRITE TESTS BEFORE RUNNING EXPERIMENTS
  I wrote 29 unit tests (pytest) covering every aggregation method, both
  detection filters, label noise, and the DP epsilon estimator. Running these
  first meant I could trust the math before running hour-long simulations.

STEP 5 — DEBUG THREE CRITICAL BUGS (the hardest part)

  Bug A — Out-of-memory crash
    Symptom: Ray actor processes crashed allocating 847 MB each.
    Fix: Switched from loading all 90k images as a torch.Tensor to numpy
    memory-mapped arrays (np.load mmap_mode="r"). The OS now shares physical
    pages across Ray processes. Each actor uses ~10 MB instead of 847 MB.

  Bug B — CPU-only PyTorch (3.7× speed penalty)
    Symptom: torch.cuda.is_available() = False despite GTX 1650 present.
    Cause: pip default install fetches the CPU-only wheel.
    Fix: pip install torch==2.6.0+cu124 --extra-index-url
    https://download.pytorch.org/whl/cu124. Round time dropped from 82s → 22s.

  Bug C — Flower CID mismatch (root cause of ALL attacks failing)
    Symptom: 0 detections, 0 dropouts, attacked model MORE accurate than clean.
    Cause: Flower ≥ 1.8 simulation assigns large random UUID integers as
    client.cid (e.g., 13236999372251260638), not partition IDs (0–9).
    My strategy was checking int(cid) in {0,1} — always False.
    Fix: The strategy now sends role ID sets as CSV strings in the FitIns
    config dict ("malicious_ids"="0,1"). Each client reads its own
    partition_id from context.node_config and self-identifies by parsing that
    CSV. Verified: after fix, partition_id=0 produced norm=680 (vs honest ~13),
    confirming the ×50 scaling attack was finally firing.

STEP 6 — RUN 6 EXPERIMENTS AND INTERPRET RESULTS
  With all bugs fixed I ran the full 6-experiment suite. Key findings:
    Exp A (clean baseline):     84.22%  — proves the model learns well
    Exp B (attack, no defence): 18.64%  — proves attacks are devastating
    Exp C (full defence):       81.41%  — proves defence recovers 97% of accuracy
    Exp D (+ differential privacy): 38.70% — reveals privacy-utility trade-off
    Exp E: non-IID (α=0.5) converges slower but still reaches ~80%
  The 7 plots in results/ tell the full story visually.

STEP 7 — WHAT I LEARNED
  - Always audit the FL framework's client identity mechanism before assuming
    integer IDs — Flower's UUID-based cid is a non-obvious footgun.
  - Memory-mapped I/O is essential when many processes share the same large file.
  - Differential privacy at noise_multiplier=1.0 (noise/clip ratio) is too
    aggressive for non-IID data — the privacy-utility curve is steep.
  - Testing aggregation math in isolation (without the FL loop) saved hours of
    debugging time during the integration phase.

TOTAL IMPLEMENTATION: ~900 lines of Python across 8 modules + 29 tests.
TOTAL RUNTIME (6 experiments on GTX 1650): ~2.5 hours.

1. PROBLEM STATEMENT
--------------------------------------------------------------------------------

Hospitals cannot share patient pathology slide images due to legal and privacy
constraints (HIPAA, GDPR). A centralised deep learning model is therefore
infeasible. Federated Learning allows hospitals to collaboratively train a shared
model without any data leaving each site.

However two real-world threats break standard Federated Learning:

  (a) Byzantine attacks — a subset of hospitals sends deliberately corrupted
      model updates to poison the global model.
  (b) Asynchronous networks — hospitals have different hardware speeds and
      network reliability, so updates arrive at different times. Standard FL
      stalls waiting for the slowest client.

This project builds a system that handles both simultaneously.


--------------------------------------------------------------------------------
2. SYSTEM ARCHITECTURE
--------------------------------------------------------------------------------

DATASET
  PathMNIST — 107,180 colon-tissue pathology patches, 28×28 RGB, 9 tissue
  classes (adipose, background, debris, lymphocytes, mucus, normal colon
  mucosa, colorectal adenocarcinoma, smooth muscle, stroma).
  Train split: 89,996 images. Test split: 7,180 images.

MODEL: PathologyNet CNN
  Input  : Tensor [B, 3, 28, 28]
  Conv1  : 3 → 32 filters, 3×3, padding=1, ReLU → MaxPool(2×2)
  Conv2  : 32 → 64 filters, 3×3, padding=1, ReLU → MaxPool(2×2)
  FC1    : 64×7×7 = 3136 → 128, ReLU, Dropout(0.25)
  FC2    : 128 → 9 logits
  Output : 9-class cross-entropy loss
  Device : NVIDIA GeForce GTX 1650 (CUDA 12.4, PyTorch 2.6.0+cu124)

FEDERATION TOPOLOGY
  10 hospitals (clients) total.
  6 hospitals sampled per round.
  4 fastest clean updates used for aggregation (async buffer).
  20 training rounds.
  Local epochs: 2 (rounds 1–3 warm-up), 5 (rounds 4–20).
  Batch size: 32.

DATA HETEROGENEITY
  Dirichlet partitioning (α = 0.5) creates non-IID distributions across
  hospitals — each hospital has a different class distribution mimicking
  real-world inter-hospital variation in patient populations.

CLIENT TAXONOMY (10 hospitals)
  Hospitals {0, 1}   — MALICIOUS (20%)
                       Attack type: gradient scaling ×50
  Hospitals {4, 5}   — NOISY (20%)
                       30% of training labels randomly flipped
  Hospitals {6, 7}   — UNRELIABLE (20%)
                       40% probability of dropout per round
  Hospitals {2,3,8,9} — HONEST (40%)


--------------------------------------------------------------------------------
3. DEFENCE MECHANISMS
--------------------------------------------------------------------------------

STEP 1 — DETECTION FILTERS (applied before async selection)
  L2 Norm Filter:
    Compute L2 norm of each flattened update.
    Reject if norm > 3 × median norm across all submitted updates.
    Rationale: scaling attack multiplies gradients by 50×, inflating norm by
    ~50× relative to honest updates.

  Cosine Similarity Filter:
    Compute coordinate-wise median of all updates as reference direction.
    Reject if cosine similarity < 0 (update points to opposite half-space).
    Rationale: sign-flip and directional poisoning attacks reverse the gradient
    direction relative to the consensus.

STEP 2 — ASYNC SELECTION
  After detection, surviving updates are sorted by simulated arrival delay
  (sampled from Exp(scale=2.0)). The 4 fastest survivors are used; the
  remaining 2 are treated as stragglers and discarded for this round.
  Design intent: attackers cannot game the async buffer by responding fast
  because detection runs before selection.

STEP 3 — ROBUST AGGREGATION
  Method: Coordinate-wise trimmed mean (Yin et al., ICML 2018).
  Trim fraction: 10% from each tail per coordinate.
  Staleness weighting: weight = 1 / (1 + staleness_rounds), so stale updates
  from slow hospitals contribute proportionally less.

STEP 4 — DIFFERENTIAL PRIVACY (Experiment D only)
  Mechanism: Client-side adaptive DP clipping (Flower built-in).
  Noise multiplier: 0.1
  Initial clipping norm: 0.1
  Delta: 1e-5
  Privacy budget computed via Rényi DP accountant: ε ≈ 0.6239
  Interpretation: any adversary's ability to distinguish a participating patient
  record from a non-participating one is bounded by e^0.6239 ≈ 1.87×.


--------------------------------------------------------------------------------
4. EXPERIMENT DESIGN
--------------------------------------------------------------------------------

  Exp A — FedAvg, no attack
           Clean non-IID baseline. No malicious clients. No detection.
           Purpose: establish upper-bound accuracy without any adversary.

  Exp B — FedAvg, under attack, no defence
           Malicious + noisy clients active. Detection filters OFF.
           Purpose: show raw vulnerability of standard FedAvg to Byzantine
           gradient scaling attacks.

  Exp C — AsyncRobustFL, under attack, with defence
           Malicious + noisy + unreliable clients active.
           Detection filters ON. Trimmed-mean aggregation.
           Purpose: show that the defence recovers accuracy lost in Exp B.

  Exp D — AsyncRobustFL + Differential Privacy
           Same as Exp C plus client-side adaptive DP noise.
           Purpose: show the privacy-utility trade-off of adding DP.

  Exp E — Heterogeneity isolation
           Two runs: α=0.5 (non-IID) vs α=1000.0 (near-IID).
           All other settings identical to Exp C.
           Purpose: isolate the effect of data heterogeneity on convergence.

7. BUG FIXES APPLIED DURING DEVELOPMENT
--------------------------------------------------------------------------------

  BUG 1 — OOM crash in Ray actor processes
    Symptom: RuntimeError: DefaultCPUAllocator: not enough memory
              trying to allocate 846,682,368 bytes
    Root cause: data.py loaded all 89,996 PathMNIST training images as a
                contiguous torch.Tensor (~847 MB) inside each Ray actor process.
                With multiple parallel actors, total RAM exceeded machine limit.
    Fix: Replaced torch.Tensor cache with numpy memory-mapped arrays
         (np.load mmap_mode="r"). OS shares physical pages across processes.
         Only each client's ~900-sample subset is converted to tensors per call.

  BUG 2 — Attacks, dropouts, and detection never triggering
    Symptom: clients_filtered=0, clients_dropped=0 across all 20 rounds.
             Attacked model more accurate than clean model (inverted narrative).
    Root cause: Flower 1.8+ simulation assigns large UUID integers as client.cid
                (e.g., 13236999372251260638), not partition IDs (0–9).
                strategy.configure_fit was checking int(client.cid) in {0, 1}
                which was always False. No client was ever marked malicious or
                unreliable.
    Verified: 2-round diagnostic script confirmed cid=13236999372251260638
              while partition_id=0 (malicious). int(cid) ∉ {0,1}.
    Fix: strategy now sends role ID sets as CSV strings in FitIns config
         ("malicious_ids"="0,1", "unreliable_ids"="6,7", "noisy_ids"="4,5").
         Each client self-identifies by checking its own partition_id against
         the parsed sets. Verified: after fix, partition_id=0 produces norm=680
         (vs honest ~13), is_malicious=True, Hospital-7 drops out correctly.

  BUG 3 — CPU-only PyTorch installed
    Symptom: torch.cuda.is_available() = False despite GTX 1650 present.
             Training ~3.7× slower than necessary.
    Root cause: pip default install fetches CPU-only torch build.
    Fix: pip install torch==2.6.0+cu124 torchvision==0.21.0+cu124
         --extra-index-url https://download.pytorch.org/whl/cu124
    Result: CUDA: True, GPU: NVIDIA GeForce GTX 1650.
            Round time reduced from ~82s (CPU) to ~22s (GPU).

  BUG 4 — Flower API signature mismatches (resolved earlier)
    run_simulation() got unexpected keyword argument 'run_config'
    → Removed run_config; replaced bare client_fn with make_client_fn()
      closure factory.
    ServerAppComponents() got unexpected keyword argument 'server_config'
    → Corrected to config=ServerConfig(num_rounds=...) via API inspection.

10. KEY DESIGN DECISIONS
--------------------------------------------------------------------------------

  Filter-before-select (not select-then-filter):
    If detection ran after async selection, a malicious client could respond
    instantly to guarantee inclusion. Running detection on all 6 submissions
    before selecting the fastest 4 removes this attack vector.

  Staleness weighting:
    weight = 1 / (1 + Δversion) penalises stale updates without discarding
    them. A client 3 rounds stale gets weight 0.25 vs weight 1.0 for fresh.

  Coordinate-wise median as cosine reference:
    Using the mean as cosine reference is self-defeating under attack because
    malicious updates shift the mean toward the attack direction. The median
    is robust as long as fewer than 50% of clients are Byzantine (here 20%).

  Memory-mapped data loading:
    PathMNIST stored as npz. Each Ray actor memory-maps the file with
    mmap_mode="r" so the OS shares physical pages. Peak per-actor RAM is ~10 MB
    (one client's slice) instead of 847 MB (full dataset).
